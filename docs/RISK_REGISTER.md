# Part 2 Risk Register

| ID | Risk | Likelihood | Impact | Mitigation | Review Status |
|---|---|---:|---:|---|---|
| R-01 | Two-day schedule is consumed by framework integration | Medium | High | Build a fixture-backed vertical slice first; keep Haystack components thin | Open |
| R-02 | Cloud-only demo conflicts with local open-model wording | High | High | Verify Docker Ollama/Qwen path and disclose Demo mode | Resolved in P7: the hosted demo is deterministic-only (no cloud model provider at all, D-05 stays open), so the conflict this risk described cannot occur — there is no cloud model path to disclose. The local Docker Ollama/Qwen path remains verified per P3 (`docs/SECURITY_EVIDENCE.md` and `IMPLEMENTATION_PHASE_PLAN.md` P3 notes) and is presented in the hosted demo narrative as recorded evidence, not reproduced live. |
| R-03 | Lightweight local model produces weak legal classification | Medium | High | Use `qwen3:4b` first to prove workflow; fall back to a larger permissively licensed local model only if fixture quality fails; disclose limitations | Live-verified (4/4 runs identical) but risk **materialized in a bounded way**: `qwen3:4b` did not classify `confidentiality`/`ip_ownership` clause types present in the golden fixture text (see IMPLEMENTATION_PHASE_PLAN.md P3 Live Verification Notes). Repeatability and confidence-threshold criteria pass; recommend the larger-model fallback before any non-demo reliance on model-path coverage. |
| R-04 | Prompt injection in contract alters output | Medium | High | Treat content as untrusted, expose no tools, validate outputs, adversarial fixture | Deterministic path tested since P2; P3 model path live-verified against Docker Ollama + `qwen3:4b` (4/4 runs), no injected-instruction compliance observed (SYSTEM_INSTRUCTION explicitly forbids following in-document instructions) |
| R-05 | Docling model artifacts download at runtime | Medium | High | Confirmed in P1 spike (unconditional HF Hub download even with OCR disabled); mitigated for now by using `pypdf`/`python-docx` instead, which need no artifacts. Re-opens if Docling is re-adopted in P6: pre-fetch and pin artifacts, test with egress disabled | Mitigated for P1 (parser swap); re-open at P6 if Docling is re-adopted |
| R-06 | Qdrant is exposed without authentication | Medium | Critical | Private bind, API key, TLS/network policy; never publish its port | Mitigated in P4/P6: no host port published (P4); API-key authentication implemented and optional in P6 (`Settings.qdrant_api_key`, unset = Qdrant's own unauthenticated default), wired consistently through `GuidanceService`/health-check/`index_guidance.py` and verified live end to end, including a real bug found and fixed (empty-string key wrongly enforces auth in Qdrant — see `SECURITY_EVIDENCE.md` §5). TLS remains open, P7 scope. |
| R-07 | Configurable model endpoint enables SSRF | Medium | Critical | Provider allowlist, scheme/host validation, DNS/IP checks, egress policy | Mitigated in P5/P6: provider allowlist (P5); P6 adds DNS resolution + explicit blocklist (link-local/metadata/reserved/multicast, always blocked) with private RFC1918 ranges allowed only in local mode — all verified live (`SECURITY_EVIDENCE.md` §1). Not a full enterprise egress-policy allowlist; a public IP is allowed by default and DNS-rebinding-resistance is out of scope for this PoC. |
| R-08 | API key leaks through config UI or logs | Medium | Critical | Server-only secrets, boolean credential status, redaction tests | Mitigated in P5/P6: credential field write-only, never returned (P5). P6 closes the log-redaction gap specifically: `job_runner.py`'s exception handler no longer logs exception messages/tracebacks that could echo document evidence or (in principle) other sensitive strings — verified with a test forcing a failure carrying fake secret/evidence sentinels and asserting they never appear in logs (`SECURITY_EVIDENCE.md` §3). |
| R-09 | Uploaded malicious PDF exploits parser | Low/Medium | Critical | Size/type checks, patched parser, non-root isolation, timeouts | Open until tested |
| R-10 | LLM variability is presented as deterministic | Medium | Medium | Use “deterministic workflow”; record versions and test variance | Mitigated in specs |
| R-11 | License of a transitive dependency or model changes | Low/Medium | High | Pin revisions, SBOM, license scan, review final artifacts | Open until lock |
| R-12 | Public demo receives real sensitive documents | Medium | Critical | Visible warning, synthetic fixture, access control, short retention | Mitigated in P7: persistent non-dismissible frontend banner warns against real/client/classified/export-controlled/privileged uploads (`DemoModeBanner`); HTTP Basic Auth restricts hosted access (D-06, fails closed if unconfigured); 24h retention (`Settings.retention_hours_demo`, unchanged since P6) plus ephemeral hosted storage on the default Render plan bounds exposure further. Residual risk: Basic Auth is presentation-level access control, not identity/audit; a determined credentialed user could still paste sensitive text — the warning and short retention are the only remaining controls against that, matching a PoC's disclosed limitations. |
| R-13 | Vector database adds complexity without demo value | Medium | Medium | Limit it to supplemental guidance/examples; never use it to exclude applicable rules | Mitigated in P4: retrieval is strictly post-processing (`rule_engine.py` untouched, never called by retrieval code), degrades cleanly when Qdrant/the embedding model is unreachable, and never blocks a review. Live-verified: retrieval-on vs retrieval-off produce identical rule outcomes. D-09 accepted. |
| R-14 | Legal playbook (and, as of P4, its supplemental guidance corpus) is mistaken for legal advice | Medium | High | Label as illustrative and require counsel validation for production | Mitigated in specs; P4's `playbooks/guidance-v1.json` carries the same disclaimer and the UI shows it inline on every guidance panel ("Illustrative decision-support content, not legal advice") |
| R-15 | In-process PoC job is interrupted by backend restart | Medium | Medium | Mark interrupted active jobs failed on recovery; use durable queue/workers in enterprise target | Open decision D-12 |
| R-16 | English demo is mistaken for validated Indonesian legal accuracy | Medium | High | Preserve Unicode/Indonesian text, label language limits, require bilingual playbook and evaluation for production claims | Mitigated in specs |

## Rule
No Critical-impact risk may remain without a documented control and successful verification step before deployment acceptance.
