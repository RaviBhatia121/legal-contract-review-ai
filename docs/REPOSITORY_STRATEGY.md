# Part 2 Repository Strategy

## Decision
Use one monorepo rooted at `Part2/legal-contract-review-ai/`. `Part2/` is the case-study
workspace folder, not the repository; the repository root is `legal-contract-review-ai/`.
This decision is accepted (reorganized from the repository-at-`Part2/` layout after P0).

## Current Layout

```text
Part2/
  CASE STUDY - ... .pdf     (kept outside the repo, one level above Part2/)
  legal-contract-review-ai/ (repository root)
    frontend/                 React, TypeScript, and Vite
    backend/                  FastAPI, rules, and persistence
    fixtures/                 Synthetic contracts and expected results (generated by
                               scripts/generate_fixtures.py; tests/, deploy/ are created when
                               the corresponding phase needs them)
    playbooks/                Versioned machine-readable playbook assets (P2:
                               defense-services-v1.json, transcribed from
                               DEFENSE_PLAYBOOK_TEMPLATE.md; mounted read-only into the
                               backend container, see docker-compose.yml)
    scripts/                  Fixture generation and other build-time-only helpers
                               (generate_fixtures.py needs the `fixtures` extra, not part of
                               the runtime backend image)
    docs/                     Specification and review pack (all *.md except README.md)
    docker-compose.yml        Local integrated stack
    .env.example              Non-secret configuration contract
    .gitignore, .dockerignore
    README.md                 Build, run, test, and architecture entry point
```

Specification files live under `docs/`; `MASTER_INDEX.md` (`docs/MASTER_INDEX.md`) stays
the entry point and is linked from the root `README.md`. Directories not yet needed
(`tests/`, `deploy/`) are created only when the phase that needs them begins.

**P2 build-context change:** the backend Docker build context moved from `backend/` to the
repo root (`context: .` in `docker-compose.yml`, `dockerfile: backend/Dockerfile`) so the
backend image can also `COPY playbooks ./playbooks` — a sibling top-level directory is not
reachable from a `backend/`-only build context. A root `.dockerignore` was added to keep
the wider build context lean (excludes `frontend/node_modules`, `backend/.venv`, `docs/`,
etc.). `backend/app/core/config.py`'s `playbook_dir` setting defaults to `../playbooks`
(correct for local `cd backend && uvicorn ...`) and is overridden to `/app/playbooks` in
`docker-compose.yml` for the container layout.

## Boundary Rules
- `frontend/` communicates only through the versioned API; it cannot import backend code.
- `backend/` owns document processing, provider adapters, rules, persistence, and security enforcement.
- `playbooks/` contains runtime assets generated from the approved playbook specifications.
- `fixtures/` contains synthetic data only.
- `deploy/` exposes only the frontend/ingress and backend API; data and model services remain private.
- Shared contracts are generated from or validated against the backend schema rather than manually duplicated.

## Dependency Management
- Frontend dependencies use one committed JavaScript lockfile.
- Backend dependencies use one committed Python lockfile.
- Container images and model artifacts are pinned by immutable version or digest before release.
- No secret, model weight, uploaded document, local database, or runtime vector data is committed.

## Change Discipline
- One change set includes code, tests, schema updates, and affected specifications.
- API or output-contract changes must update backend validation, frontend types, tests, and documentation together.
- The monorepo uses one release identifier for the complete PoC.
- A single CI workflow runs frontend, backend, contract, security, and license checks.

## Ownership and Licensing
- The PoC repository is private unless the owner explicitly decides otherwise.
- Third-party licenses and notices are preserved and generated for delivery.
- Using permissively licensed dependencies does not automatically license the PoC source code for public distribution.
